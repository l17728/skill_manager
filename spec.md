# 提示词 & Skill 对比验证与优势重组平台 - SPEC+TODO 合并文档（SDD落地专用）

# 一、文档概述

## 1.1 目的

本文档为 **提示词 / Skill / Agent 对比、验证、关键信息识别、优势重组自动化平台** 提供完整、可直接用于开发的规格定义+落地任务清单，遵循：

- SDD（Spec-Driven Development）：先规格、后实现

- TDD（Test-Driven Development）：先定义可测试行为，后编码

- AI Coding 友好：结构清晰、条目化、可直接映射为功能模块+开发任务

## 1.2 适用范围

- 支持 Claude Skill / Agent 管理、多维度标签（含自动打标签）

- 支持测试基线（测试用例）管理

- 支持基于 **Claude CLI 客户端** 的自动化对比测试

- 支持差异分析、优势提取、自动重组、迭代验证

- 支持完整上下文管理、版本管理、日志与文件归档

## 1.3 核心约束（不可变更）

- 所有模型交互 **仅通过 Claude CLI 完成**（含自动标签生成），不直接调用 API

- 所有数据、配置、会话、日志均落地为文件/目录结构，不依赖外部数据库

- 支持长期迭代、可复现、可回溯

- 自动标签生成过程需纳入会话隔离与上下文管理体系，默认待审核后生效

# 二、产品定位与核心价值

## 2.1 定位

面向提示词工程师 / Agent 开发者的 **Skill 验证与优化工作台**，自动化完成：`管理 → 测试 → 对比 → 分析 → 重组 → 迭代 → 定稿`

## 2.2 核心价值

- 统一管理多来源 Skill / Agent（同目的、不同提供方可有序隔离）

- 客观、可复现、可对比的测试结果

- 自动识别“为什么某个 Skill 更好”

- 自动融合最优片段，生成更强提示词

- 完整可追溯：版本、模型、CLI、上下文、结果

# 三、总体架构

## 3.1 四层架构

1. **资产层**：Skill / Agent / 测试基线管理

2. **执行层**：Claude CLI 封装、进程、会话、上下文管理

3. **测试层**：对比测试、自动化执行、结果采集

4. **分析层**：差异判断、优势提取、自动重组、迭代验证

## 3.2 目录结构规范（所有数据文件化）

```Plain Text
软件文件组织规范（适配多目的、多提供方资产管理）workspace/                  # 根目录，所有资产统一归档
  projects/             # 测试项目目录（按项目独立隔离）
    project_xxx/        # 单个测试项目（命名格式：project_项目名称_时间戳）
      skills/           # 该项目引用的Skill/Agent副本（避免原资产修改影响项目）
      baselines/        # 该项目引用的测试基线副本
      .claude/          # 项目专属会话目录（独立隔离，不污染其他项目）
      config.json       # 项目配置（模型版本、CLI参数等）
      logs/             # 项目专属日志
  skills/               # Skill / Agent 主库（按“目的+提供方”分层组织）
    功能目的分类/       # 一级目录：按Skill/Agent核心目的分类（如text_summary、code_generate）
      提供方标识/       # 二级目录：按提供方分层（如provider_A、provider_B、internal）
        skill_xxx_v1/   # 单个Skill/Agent版本目录（含版本号，支持多版本共存）
          content.txt   # Skill/Agent核心提示词内容
          meta.json     # 元数据（名称、描述、作者、版本、标签等）
          tags.json     # 标签文件（人工标签+已生效自动标签）
          auto_tag_log/ # 自动打标签生成记录（CLI日志、会话路径等）
          history/      # 版本历史（编辑前后diff、修改时间）
  baselines/            # 测试基线主库（与Skill组织逻辑一致）
    功能目的分类/       # 一级目录：按基线测试目的分类（如text_summary_test、code_generate_test）
      提供方标识/       # 二级目录：按提供方分层（如provider_A、provider_B、internal）
        baseline_xxx_v1/# 单个基线版本目录（含版本号）
          cases.json    # 测试用例集合（用例ID、输入、预期输出等）
          meta.json     # 基线元数据
          tags.json     # 基线标签文件
          auto_tag_log/ # 基线自动打标签记录
          history/      # 基线版本历史
  cli/                  # CLI 配置、缓存、临时会话
    config.json         # CLI全局配置（路径、默认模型、超时等）
    temp_session/       # 自动打标签专用临时会话（默认保留7天，过期清理）
    cache/              # CLI执行缓存
  logs/                 # 系统全量日志（CLI执行、自动打标签、编辑操作等）
  versions/             # 全局版本快照（定期自动备份，支持回滚）
```

## 3.3 技术栈

- **运行形态**：基于 **Electron** 的本地桌面应用，内嵌 Chromium 渲染 HTML/CSS/JS 界面，无需外部浏览器，双击即启动
- **后端**：Node.js（Electron 主进程），负责文件读写、子进程管理、业务逻辑
- **前端**：HTML / CSS / JavaScript（Electron 渲染进程），实现四栏 UI 布局
- **模型交互**：所有模型调用通过 **Claude Code CLI**（`claude` 命令）以 Node.js `child_process` 子进程方式完成，不直接调用 API
- **数据存储**：纯文件化存储（JSON / Markdown / 纯文本），无外部数据库依赖；`workspace/` 根目录位于项目代码库内（与源代码同仓库管理）
- **打包**：`electron-builder` 构建为 Windows `.exe` 安装包

# 六、非功能规格（NFR）

## 6.1 稳定性

- 单个用例异常不崩溃

- 支持断点续跑

- 所有操作可回滚

- 自动打标签的临时会话故障，不影响任何测试项目会话

- 批量生成任务中，单个资产失败，不中断整个批量任务的执行

## 6.2 可扩展性

- 可接入其他模型 CLI

- 可新增评价维度

- 可自定义重组策略

## 6.3 文件优先

- 不依赖外部数据库

- 所有内容可 git 管理

- 文件组织遵循“目的分类+提供方分层”原则，同目的、不同提供方的Skill/Agent、测试基线独立存储，每个资产对应独立版本目录，支持多版本共存，便于分类管理和追溯

## 6.4 性能

- 单个资产自动标签生成响应时间 ≤ 60 秒（依赖 CLI 与网络状态；实际约 15-25 秒，超时阈值设为 60 秒）

- 批量生成任务支持“后台执行”，不阻塞前端操作

## 6.5 存储

- 临时会话文件默认保留 7 天，可在配置中调整保留时长

- 自动标签生成历史记录永久保存，支持导出为 CSV
